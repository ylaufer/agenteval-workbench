# Feature Specification: Streamlit UI for AgentEval Workbench

**Feature Branch**: `002-streamlit-ui`
**Created**: 2026-03-22
**Status**: Draft
**Input**: User description: "Build a Streamlit UI for AgentEval Workbench. The UI must allow a user to enter a prompt, generate a complete demo case, validate the generated dataset case, run the existing evaluation pipeline, inspect the resulting trace and evaluation report. The UI must reuse existing library code from src/agenteval and must not duplicate business logic. The UI should support local file-based workflows only for the first version. The UI should display validation errors and evaluation results in a clear and actionable way."

## Clarifications

### Session 2026-03-22

- Q: Should the UI include a free-text prompt input field for case generation? → A: No. The UI collects only case ID and failure type; prompt content is auto-generated by the generator based on the selected failure type.
- Q: After generating a case, should validation run against only the new case or the entire dataset? → A: Validate the full dataset, but visually separate new-case issues (prominent) from other-case issues (collapsed).
- Q: How should the user navigate between the four workflow areas (Generate, Evaluate, Inspect, Report)? → A: Sidebar navigation with separate pages. Each page operates independently.
- Q: Should the UI offer a standalone "Validate Dataset" action outside the generation workflow? → A: Yes, add a "Validate Dataset" button on the Generate page alongside auto-validation after generation.
- Q: Which capabilities are explicitly excluded from v1 beyond score editing and calibration? → A: Case deletion, in-place file editing, custom rubric selection, and trace upload from external files are all out of scope.

## User Scenarios & Testing

### User Story 1 - Generate and Validate a Demo Case (Priority: P1)

A user opens the Streamlit UI and enters an optional case ID (auto-generated if left blank) and optionally selects a failure type from the 12 canonical failure categories. They click a "Generate Case" button, and the system creates a complete benchmark case (prompt.txt, trace.json, expected_outcome.md) in the local dataset directory. The prompt content is auto-generated by the generator based on the selected failure type — the UI does not accept free-text prompt input. After generation, the system automatically validates the new case and displays the validation result — either a success confirmation or a list of actionable error/warning messages with severity levels and affected files.

**Why this priority**: This is the core value proposition — enabling non-CLI users to create and validate benchmark cases through a visual interface. Without this, the UI provides no standalone value.

**Independent Test**: Generate a case with a selected failure type, verify the 3 files are created on disk, and confirm validation results are displayed with pass/fail status and any issues listed by severity.

**Acceptance Scenarios**:

1. **Given** the UI is open with an empty form, **When** the user enters a case ID and clicks "Generate Case", **Then** a case directory is created with prompt.txt, trace.json, and expected_outcome.md, and a success message is displayed.
2. **Given** the UI is open, **When** the user selects "Tool Hallucination" as the failure type and generates a case, **Then** the expected_outcome.md header contains `Primary Failure: Tool Hallucination` and `Severity: Critical`.
3. **Given** a case was just generated, **When** validation runs automatically, **Then** the full dataset is validated and the result is displayed showing pass/fail status. Issues from the newly generated case are shown prominently at the top; issues from other cases are shown in a separate collapsed section below.
4. **Given** the user tries to generate a case with an ID that already exists, **When** overwrite is not enabled, **Then** an error message is displayed explaining the conflict and offering the overwrite option.

---

### User Story 2 - Run Evaluation Pipeline on Cases (Priority: P2)

A user selects which cases to evaluate (or evaluates all cases in the dataset). They click "Run Evaluation" and the system generates structured evaluation templates (JSON and Markdown) for each case using the existing rubric. The UI displays a summary of the evaluation run showing how many cases were processed, the rubric dimensions scored, and any issues encountered. The user can then browse individual case evaluation templates.

**Why this priority**: Evaluation is the next logical step after case generation. It transforms raw cases into structured assessment templates that reviewers can use.

**Independent Test**: With at least one valid case in the dataset, run the evaluation pipeline and verify that evaluation template files are created in the reports directory, and the UI displays the evaluation summary.

**Acceptance Scenarios**:

1. **Given** valid cases exist in the dataset, **When** the user clicks "Run Evaluation", **Then** evaluation templates (.evaluation.json and .evaluation.md) are generated in the reports directory for each case.
2. **Given** the evaluation pipeline has run, **When** results are displayed, **Then** the user sees a per-case summary including case ID, primary failure type, severity, and dimension scores (or "unscored" placeholders).
3. **Given** a case has an invalid trace (fails schema validation), **When** the evaluation pipeline runs, **Then** the error is reported in the UI without crashing the entire run.

---

### User Story 3 - Inspect Trace and Evaluation Details (Priority: P2)

A user browses the list of existing cases and selects one to inspect. The UI displays the trace steps in a readable format (step type, actor, content, tool calls and outputs). Below the trace, the evaluation template is shown with each rubric dimension, its scale, weight, scoring guide, and current score (if filled). The user can also view the raw expected_outcome.md header metadata (failure type, severity, case version).

**Why this priority**: Inspection is essential for reviewers to understand what the agent did and how it was evaluated. It complements generation and evaluation by closing the feedback loop.

**Independent Test**: Select an existing case, verify the trace steps render correctly with all fields visible, and confirm the evaluation template (if generated) is displayed alongside it.

**Acceptance Scenarios**:

1. **Given** a case exists with a valid trace, **When** the user selects it for inspection, **Then** the trace steps are displayed in order showing step_id, type, actor_id, and content for each step.
2. **Given** a case has tool_call steps, **When** inspecting the trace, **Then** tool_name, tool_input, and corresponding tool_output (from the next observation step) are displayed.
3. **Given** an evaluation template exists for a case, **When** inspecting the case, **Then** each rubric dimension is shown with its title, scale, weight, and current score or "Not yet scored" placeholder.
4. **Given** a case has expected_outcome.md with a YAML header, **When** inspecting the case, **Then** the header metadata (Case ID, Primary Failure, Secondary Failures, Severity, case_version) is displayed.

---

### User Story 4 - View Aggregated Evaluation Report (Priority: P3)

A user clicks "Generate Report" after evaluation templates have been filled in (or partially filled). The system aggregates all evaluation templates into a summary report showing per-dimension statistics (mean scores, distributions), overall normalized scores per case, failure type frequency, and improvement recommendations. The report is displayed in the UI and also saved as files (JSON and Markdown).

**Why this priority**: Aggregation provides the high-level insights that stakeholders need. It depends on evaluation templates being generated and ideally scored, making it a later-stage workflow.

**Independent Test**: With at least two cases that have evaluation templates, generate the summary report and verify dimension statistics and failure distributions are displayed.

**Acceptance Scenarios**:

1. **Given** multiple evaluation templates exist, **When** the user clicks "Generate Report", **Then** a summary is displayed with per-dimension mean scores and score distributions.
2. **Given** the report has been generated, **When** viewing it, **Then** the user sees primary failure frequency counts and severity distribution across all cases.
3. **Given** the report is generated, **When** viewing it, **Then** summary.evaluation.json and summary.evaluation.md are saved to the reports directory.

---

### Edge Cases

- What happens when no cases exist in the dataset directory? The UI displays an informational message and guides the user to generate a case first.
- What happens when the reports directory does not exist? The system creates it automatically when writing evaluation outputs.
- What happens when the rubric file is missing or invalid? The evaluation pipeline displays a clear error message identifying the missing/invalid rubric file path.
- What happens when a trace.json fails schema validation during evaluation? The error is reported for that specific case; other cases continue processing.
- What happens when the user generates a case but the dataset directory is outside the repo root? The system displays the security error from the validator without exposing internal paths.

## Requirements

### Functional Requirements

- **FR-001**: The UI MUST provide a form to enter a case ID (optional, auto-generated if blank) and select an optional failure type from the 12 canonical categories. The UI MUST NOT include a free-text prompt field — prompt content is auto-generated by the generator based on the selected failure type.
- **FR-002**: The UI MUST generate a complete case by calling the existing `generate_case()` library function, not by reimplementing case creation logic.
- **FR-003**: The UI MUST validate the full dataset by calling the existing `validate_dataset()` library function after case generation. Issues from the newly generated case MUST be displayed prominently; issues from other cases MUST be shown in a separate collapsed section. The Generate page MUST also provide a standalone "Validate Dataset" button that runs validation independently of case generation.
- **FR-004**: The UI MUST display validation issues grouped by severity (errors first, then warnings) with the affected case ID and file path for each issue.
- **FR-005**: The UI MUST run the evaluation pipeline by invoking the existing runner module, producing JSON and Markdown evaluation templates.
- **FR-006**: The UI MUST display trace steps in a structured, readable format showing step sequence, type, actor, content, and tool interaction details.
- **FR-007**: The UI MUST display evaluation templates with rubric dimension details (title, scale, weight, scoring guide, current score).
- **FR-008**: The UI MUST generate aggregated summary reports by invoking the existing report module.
- **FR-009**: The UI MUST NOT duplicate any business logic from `src/agenteval` — all case generation, validation, evaluation, and reporting MUST delegate to existing library functions.
- **FR-010**: The UI MUST operate entirely with local files — no network calls, no external services, no cloud storage.
- **FR-011**: The UI MUST respect the repository root boundary — all file operations MUST stay within the repo root using existing path safety mechanisms.
- **FR-012**: The UI MUST display clear, actionable error messages when operations fail, including the specific error and suggested remediation.
- **FR-013**: The UI MUST allow the user to select an overwrite option when generating a case with an existing case ID.
- **FR-014**: The UI MUST list all existing cases in the dataset directory for browsing and inspection.
- **FR-015**: The UI MUST use sidebar navigation with separate pages for each workflow area: Generate, Evaluate, Inspect, and Report. Each page operates independently and the user can navigate directly to any page at any time.

### Key Entities

- **Benchmark Case**: A directory containing prompt.txt, trace.json, and expected_outcome.md. Identified by case ID (directory name). Has a failure type, severity, and case version.
- **Validation Result**: The outcome of dataset validation — a pass/fail flag and a collection of issues, each with a case ID, file path, message, and severity level (error or warning).
- **Evaluation Template**: A structured assessment of a single case against the rubric — contains dimension scores, trace summary, failure metadata, and reviewer notes. Exists as both JSON and Markdown.
- **Summary Report**: An aggregation of all evaluation templates — contains per-dimension statistics, normalized scores, failure distributions, and improvement recommendations.

## Success Criteria

### Measurable Outcomes

- **SC-001**: A user with no CLI experience can generate a valid benchmark case, validate it, and view the validation result within 2 minutes of opening the UI.
- **SC-002**: All 12 canonical failure types are selectable from the UI and produce correctly tagged cases when generated.
- **SC-003**: Validation errors and warnings are displayed with sufficient context (case ID, file, message, severity) for the user to identify and fix the issue without consulting documentation.
- **SC-004**: The evaluation pipeline processes all valid cases in the dataset and produces evaluation templates without requiring manual file manipulation.
- **SC-005**: Trace inspection displays all step types (thought, tool_call, observation, final_answer) with their complete metadata in a readable format.
- **SC-006**: The UI reuses 100% of existing library functions for business logic — zero lines of duplicated validation, generation, evaluation, or reporting code.
- **SC-007**: All file operations remain within the repository root boundary, consistent with the project's security-first principle.

## Assumptions

- The user has the project installed in editable mode (`pip install -e ".[dev]"`) with Streamlit available as an additional dependency.
- The user runs the UI locally on their development machine with access to the repository filesystem.
- The rubric file (`rubrics/v1_agent_general.json`) and schema files (`schemas/`) are present in the repository.
- The first version does not support concurrent multi-user access — it is a single-user local tool.
- Score editing (filling in dimension scores in evaluation templates) is out of scope for the first version — the UI displays scores if present but does not provide an editing interface.
- Inter-reviewer calibration analysis is out of scope for the first version.
- Case deletion is out of scope for the first version — users must delete case directories manually.
- In-place file editing (modifying prompt.txt, trace.json, or expected_outcome.md from the UI) is out of scope for the first version.
- Custom rubric selection is out of scope for the first version — the UI uses the default rubric (`rubrics/v1_agent_general.json`).
- Trace upload from external files is out of scope for the first version — cases are generated via the built-in generator only.
